# kb-foundry · 知识铸造厂

> *「把你的业务熔铸成 AI 能跑起来的操作系统」*

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)
[![Agent Skills](https://img.shields.io/badge/Agent%20Skills-Standard-green)](https://agentskills.io)

<br>

**诊断、搭建、陪跑——帮客户把模糊业务铸成能用的知识库系统。**

不是给你一套固定目录模板让你自己填。<br>
是你告诉我客户做什么的，我帮你诊断、搭架构、带你把第一条业务流跑通、告诉你还缺什么能力、怎么补。

## 效果示例

一个做企业培训的客户说「我想整理一下我的课程和学员资料」。

```
用户      > 我是做企业培训的，课程资料、学员反馈、方案文档散在各处，
            想整理但不知道从哪开始。

kb-foundry > 先不急着整理。我先帮你把业务看清楚。
            你的课卖给谁？是 to B 还是 to C？收入主要来自课程费还是咨询费？

用户      > to B，主要是课程费，偶尔也做培训咨询。

kb-foundry > 明白了。你的核心业务流大概是：
            客户需求 → 方案 → 课程交付 → 学员反馈 → 复盘优化。
            我建议先用 Standard 规模，搭 3 条业务流。
            说「保存」我就帮你生成目录骨架和 AI 执行规则。

用户      > 保存。

kb-foundry > 已生成。你的知识库现在有 12 个目录，AGENTS.md 定义了
            AI 能帮你做什么、什么必须你确认。
            现在带你跑通第一条业务流——假设刚收到一个客户培训需求。
            你把这个需求写进 02_Input_Inbox/，然后我们一步步走。
```

15 分钟后，客户手里有一套能用的知识库：目录、AI 执行规则、业务流开关、能力开关、工作台，以及「我知道下一个需求来了该怎么处理」的信心。

再看一个内容创作者的场景：

```
用户      > 我是做自媒体的，每天要写公众号、发朋友圈、偶尔做视频。
            选题靠灵感，写完就丢了，想系统化但不知道怎么搭。

kb-foundry > 你的核心业务流是：输入收集 → 选题决策 → 写作成稿 → 发布分发 → 数据复盘。
            内容型业务我建议 Lite 起步，先搭 2 条核心流。
            你平时灵感从哪来？

用户      > 刷推、看书、跟人聊天，有时候洗澡想到的。

kb-foundry > 好。所有灵感先扔 02_Input_Inbox/，不用分类。
            每周做一次选题决策时，从收件箱里挑。
            说「保存」我帮你生成目录。

用户      > 保存。然后呢？

kb-foundry > 目录已生成。现在带你走一遍——
            假设你刚刷到一条有意思的推，把它写进收件箱。
            然后我们一步步看它怎么变成一篇公众号文章。
```

## 装上就能用

打开你正在用的 AI agent，告诉它：

```
帮我安装这个 skill：https://github.com/ghostyee2023/kb-foundry
```

或者用通用 CLI 安装器：

```bash
npx skills add ghostyee2023/kb-foundry
```

装好后直接说话：

```
帮我为一个做法律咨询的客户设计知识库
我是做内容的，帮我搭一套选题到发布的工作流
新手引导，第一次用
目录搭好了，带我跑一遍业务流
```

如果自动加载失败，手动把 `SKILL.md` 的内容粘贴进对话也能用——它本质就是一份 markdown + YAML frontmatter。

## 能做什么

kb-foundry 分三个阶段、七个步骤，从模糊需求到能跑起来的系统：

| 阶段 | 步骤 | 产出 |
| --- | --- | --- |
| **诊断搭建** | ① 识别业务类型 → ② 业务分析 → ③ 生成架构文件 | 目录骨架、AGENTS.md、业务流开关表、能力开关表、工作台 |
| **走查陪跑** | ④ 首条业务流走查 → ⑤ 首日运行 | 用户亲手跑通第一条完整业务流，15 分钟上手 |
| **交付验收** | ⑥ 验收检查 → ⑦ 输出交付材料 | 搭建说明、培训路径、已知卡点、升级建议 |

### 三种交付规模

不是所有客户都需要完整系统。先判断规模，再动手：

| 规模 | 适合谁 | 产出重点 |
| --- | --- | --- |
| **Lite 轻量版** | 个人、小团队、试点 | 1-2 条核心业务流、简化目录、少量模板 |
| **Standard 标准版** | 有稳定业务、想引入 AI 协作 | 完整开关表、AI 执行规则、首批模板 |
| **Full 完整交付** | 咨询交付、组织级项目 | 项目级 skill 规划、培训路径、验收记录 |

### 客户拿到手第一天就能用

- **工作台** — 打开知识库就知道今天该干什么，不用翻目录
- **本地搜索** — `python hybrid_search.py "关键词" --root .` 即搜即得，中英文混合
- **端到端样例** — `examples/lawyer-firm/` 一个完整的律师知识库，照着改比从零开始快十倍

## 怎么做的

kb-foundry 不是给你一套固定模板让你自己填。它像一个教练，按这个顺序带着你走：

**先看清楚** — 你的业务到底在做什么、钱从哪来、什么东西反复出现、什么东西丢了最疼。

**再搭骨架** — 根据你的业务类型（内容、咨询、培训、团队协作）生成对应的目录结构，用你自己的业务语言命名，不是照搬别人的模板。

**然后跑起来** — 搭好不是结束，是开始。选一条你最常用的业务流，用一个真实输入从头跑到尾：东西从哪来、放哪、谁来判断、怎么处理、交付给谁、怎么复盘。

**最后补能力** — 跑通之后你自然知道缺什么：缺模板就补模板，缺 AI 辅助就装对应的 skill，缺规则就定规则。不是一开始就塞一堆你用不上的东西。

### 不只是目录，是引擎

客户知识库搭好后，`11_能力与工具/` 下面不是空壳。kb-foundry 会生成四份文件，告诉用户：

- **你现在有什么能力** — 哪些环节 AI 能帮你，哪些必须你自己判断
- **建议装什么通用工具** — 文档处理、表格、搜索等，按你的行业推荐，你自己按清单装
- **以后可以开发什么专属能力** — 先跑通业务流，连续跑通 3 次再考虑封装成自动化
- **怎么装、怎么验证、出问题怎么回退** — 不是装完就不管了

## 适合谁

- **内容/IP 型业务** — 选题、写作、发布、数据复盘、方法论复用
- **咨询/服务型业务** — 客户诊断、方案生成、项目交付、案例沉淀、SOP 升级
- **课程/培训型业务** — 课程设计、教学资产、学员输入、答疑复用、班级复盘
- **内部团队知识库** — 角色协作、项目资料、决策记录、交接文档、治理规则
- **AI 转型项目** — 场景发现、流程重构、能力配置、工具落地、培训和迭代证据

如果你只是做一个个人读书笔记库，kb-foundry 可能过重。用 Lite 版，只保留诊断摘要和核心目录就好。

## 局限

诚实地说：

- **beta 阶段** — 核心方法链路已验证，但缺少多行业真实客户端到端交付证据。
- **不执行搭建** — kb-foundry 是设计教练，实际的文件创建依赖 AI 的通用能力。如果 AI 本身对文件操作不熟练，体验会打折扣。
- **目前只有一个行业示例** — lawyer-firm 做得很细，但其他行业需要从零翻译。
- **Full 版对个人过重** — 如果你只是整理个人资料，用 Lite 版，不要上 Full。
- **平台兼容性** — 不同 AI runtime 对 skill 的加载方式不同，如果自动加载失败，手动读取 SKILL.md 也能用。

一个不告诉你局限在哪的 skill，不值得信任。
